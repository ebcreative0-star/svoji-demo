## Summary

NLP.js (the `@nlpjs/*` monorepo) is a mature Node.js NLP library with native Czech language support via `@nlpjs/lang-cs` (tokenizer + stemmer). The library is stable at v4.26.x (stable channel); v5.0.0-alpha.5 exists but is not production-ready. For this phase the recommended package is `@nlpjs/basic` + `@nlpjs/lang-cs`, which gives full intent classification with Czech stemming at ~91KB combined unpacked size.

The integration pattern is a module-level singleton with global variable fallback for dev-mode hot reload. Training time for ~200 utterances is well under 1 second. Classification runs fully in-memory, easily within the 50ms latency target. Entity extraction for amounts uses NLP.js regex entities; name/title extraction can be done via trim entities or post-processing the matched utterance params.

**Primary recommendation:** Use `@nlpjs/basic@4.27.0` + `@nlpjs/lang-cs@4.26.1`, train on cold start, cache result in a module-level + `global` singleton, replace keyword fallback entirely with training utterances, and keep the existing Haiku path as confidence < 0.5 fallback.

## Architecture Patterns

### Recommended Project Structure

### Pattern 1: Module-Level Singleton with Global Fallback
**What:** Initialize and train the NLP.js manager once per process; reuse across all API route calls
**When to use:** Always -- NLP.js training is synchronous and should not run per-request

```typescript
// src/lib/ai/nlp-classifier.ts
import { dockStart } from '@nlpjs/basic';
import { LangCs } from '@nlpjs/lang-cs';
import trainingData from './training-data.json';

// Global var survives dev hot reload; in production module cache is sufficient
declare global {
  var _nlpClassifier: NlpClassifier | undefined;
}

class NlpClassifier {
  private nlp: any;
  private ready = false;

  async init() {
    const dock = await dockStart({ use: ['Basic'] });
    this.nlp = dock.get('nlp');
    this.nlp.container.use(LangCs);
    this.nlp.addLanguage('cs');
    await this.nlp.addCorpus(trainingData);
    await this.nlp.train();
    this.ready = true;
  }

  async classify(text: string): Promise<{ intent: string; confidence: number; entities: any[] }> {
    if (!this.ready) await this.init();
    const result = await this.nlp.process('cs', text);
    return {
      intent: result.intent,
      confidence: result.score,
      entities: result.entities ?? [],
    };
  }
}

export function getNlpClassifier(): NlpClassifier {
  if (!global._nlpClassifier) {
    global._nlpClassifier = new NlpClassifier();
  }
  return global._nlpClassifier;
}
```

### Pattern 4: Integrating NLP.js into classifyIntent()
**What:** Modify existing function to use NLP.js first, Haiku as fallback

```typescript
// In intent-classifier.ts
export async function classifyIntent(
  message: string,
  conversationContext?: string[]
): Promise<IntentResult> {
  // 1. Try NLP.js first
  const nlp = getNlpClassifier();
  const nlpResult = await nlp.classify(message);

  if (nlpResult.confidence >= 0.5) {
    // Extract params from entities + message text
    const params = extractParams(nlpResult.intent, nlpResult.entities, message);
    return { intent: nlpResult.intent, confidence: nlpResult.score, params };
  }

  // 2. Fallback to Haiku when NLP.js is uncertain
  return classifyWithHaiku(message, conversationContext);
}
```

### Anti-Patterns to Avoid
- **Training per request:** Never call `nlp.train()` inside the API route handler. It runs ~100-500ms and serializes all requests. Train once on init.
- **Using builtin entities (Microsoft Recognizers):** NLP.js builtin entities are slow and crash on some locales. Use regex entities or post-process text instead.
- **Relying on NLP.js for diacritics without lang-cs:** Without `@nlpjs/lang-cs`, Czech stemming falls back to basic tokenizer which won't handle "fotografovi" -> "fotograf" correctly.
- **Keeping keyword fallback as separate layer:** Keyword patterns should become NLP.js training utterances. A separate regex layer is redundant once NLP.js trains on the same patterns.
- **v5.0.0-alpha.5:** Do not use. API is unstable, last published 2025-01-23 with "alpha" tag. Stick to v4.27.0 / v4.26.1.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Czech stemming | Custom regex stem rules | @nlpjs/lang-cs StemmerCs | Czech morphology is complex (7 cases, verb conjugation); stemmer handles fotografa/fotografovi/fotografem -> fotograf |
| Confidence scoring | Custom cosine similarity | NLP.js built-in | NLP.js uses Logistic Regression + TF-IDF; hand-rolled classifiers won't match quality for 200 examples |
| Model caching | File-based serialization | In-memory module singleton | For 200 utterances, retraining is ~200ms max; simpler than serializing/deserializing a binary model |
| Training data variants | Exhaustive manual enumeration | NLP.js `composeFromPattern()` | Generates utterances from "[zaplatil|uhradil] jsem [jméno] [suma]" patterns automatically |

**Key insight:** Czech inflection (noun cases, verb forms) makes regex-based classification brittle. NLP.js with Czech stemmer normalizes inflected forms before matching, which is exactly the problem that causes Haiku misclassifications.

## Common Pitfalls

### Pitfall 1: NLP.js result.score vs result.confidence
**What goes wrong:** The process result object exposes `.score` not `.confidence`. Accessing `.confidence` returns undefined, causing the 0.5 threshold check to always pass.
**Why it happens:** NLP.js v4 property naming differs from IntentResult interface.
**How to avoid:** Map `result.score` to `confidence` in the wrapper function.
**Warning signs:** Every message passes the 0.5 threshold even for clearly uncertain inputs.

### Pitfall 2: Czech diacritics in training data
**What goes wrong:** Training utterances with diacritics ("přidej datum") may not match runtime input without diacritics ("pridej datum") if stemmer isn't applied correctly.
**Why it happens:** NLP.js lang-cs stemmer handles NFD normalization, but inconsistency in training data vs runtime input can confuse the classifier.
**How to avoid:** Include both diacritics and non-diacritics variants in training data (e.g., "přidej datum k fotografovi" AND "pridej datum k fotografovi"). The stemmer normalizes both.
**Warning signs:** Test utterances with diacritics fail while identical utterances without diacritics pass.

### Pitfall 3: NLP.js cold start on first API request
**What goes wrong:** The first classifyIntent() call after server restart takes 200-500ms because NLP.js trains synchronously.
**Why it happens:** Lazy init defers training to first request.
**How to avoid:** Trigger `getNlpClassifier().init()` in a warming function called at Next.js startup (e.g., in `instrumentation.ts` or a module imported at build time). Or accept the cold start since it only happens once per deploy.
**Warning signs:** First chat message after deployment is noticeably slow; subsequent messages are fast.

### Pitfall 4: @nlpjs/basic imports in Next.js App Router
**What goes wrong:** NLP.js may trigger a "this module cannot be used in an Edge Runtime" error if accidentally bundled for edge or client.
**Why it happens:** @nlpjs packages use Node.js APIs (fs, path); not compatible with Edge Runtime.
**How to avoid:** Ensure `nlp-classifier.ts` is only imported in server-only paths (API routes, Server Actions). Add `import 'server-only'` at the top of nlp-classifier.ts.
**Warning signs:** Build error mentioning Edge Runtime or "Dynamic server usage" on a client component.

### Pitfall 5: Intent names mismatch
**What goes wrong:** NLP.js returns intent names that don't match what action-executor expects.
**Why it happens:** Training corpus intent names drift from the string literals in isActionIntent() / isQueryIntent().
**How to avoid:** Use the exact intent names from IntentResult (checklist_update, budget_mark_paid, etc.) as intent keys in training-data.json. No mapping layer.
**Warning signs:** Actions don't execute even though NLP.js classifies correctly.

## Open Questions

1. **Entity extraction completeness**
   - What we know: NLP.js regex entities can extract amounts; trim entities can extract name/title
   - What's unclear: How reliable is entity extraction for Czech sentences with varied word order? "zaplatil jsem kameramana 15000" vs "15000 zaplateno za kameramana"
   - Recommendation: Keep amount/name extraction as a post-processing step on `result.utterance` using existing number-parsing logic from action-executor. NLP.js entities are supplementary, not primary.

2. **Training time for 200 Czech utterances**
   - What we know: NLP.js trains in < 1s for typical bots (DEV Community benchmark for ~100 utterances); Czech stemmer adds minimal overhead
   - What's unclear: Exact wall-clock time in Next.js server environment on this machine
   - Recommendation: Measure actual cold start time in Wave 1 task. If > 500ms, add instrumentation.ts warming. If < 300ms, accept it.

3. **NLP.js score distribution for short Czech utterances**
   - What we know: The 0.5 threshold triggers Haiku fallback
   - What's unclear: Whether NLP.js scores very short messages (e.g., "OK", "díky") above 0.5 on a wrong intent
   - Recommendation: Add explicit training utterances for small_talk and advice_request intents so low-confidence "other" messages score below 0.5 and route correctly to Haiku.
