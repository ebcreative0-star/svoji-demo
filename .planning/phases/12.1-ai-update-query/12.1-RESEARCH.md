## Summary

Phase 12.1 extends the established Classifier -> action-executor -> system prompt injection pipeline with 8 new intents and one new UI component. The codebase is clean and well-structured: every new requirement maps directly onto an existing file with a clear extension pattern.

The two hardest problems are (1) routing query intents through the action-executor so they return data the route.ts can inject into the system prompt, and (2) the global search modal which needs access to all three data domains simultaneously. Query intents are a natural extension: they return an `ActionResult` with `data` containing the formatted context string instead of mutating the DB. The search modal is the only genuinely new component; it needs data from checklist, budget, and guests which currently each live in separate page.tsx server components.

The tag and category awareness work (INTENT-07, INTENT-08, INTENT-09) is purely classifier prompt engineering -- no schema changes needed since the `tags text[]` columns from migration 009 already exist on all three tables.

**Primary recommendation:** Implement in three isolated sub-tasks: (1) update intents + classifier, (2) query intents + data injection, (3) search modal. Each can ship and test independently.

### Pattern 1: Query Intent as ActionResult with data string

Query intents (INTENT-03 through INTENT-06) go through the same `executeAction()` switch/case but return data for system prompt injection instead of mutating the DB. The `data` field carries a formatted context string.

```typescript
// In action-executor.ts -- new query handler
async function queryChecklist(
  supabase: SupabaseClient,
  coupleId: string,
  params: { filter?: 'overdue' | 'this_month' | 'all' }
): Promise<ActionResult> {
  const { data: items } = await supabase
    .from('checklist_items')
    .select('title, completed, due_date, priority, tags')
    .eq('couple_id', coupleId)
    .order('due_date', { ascending: true });

  // Format as plain text context block
  const pending = items?.filter(i => !i.completed) ?? [];
  const done = items?.filter(i => i.completed) ?? [];
  const context = `Checklist: ${pending.length} nesplněných, ${done.length} hotových.\n`
    + pending.slice(0, 10).map(i =>
        `- ${i.title}${i.due_date ? ` (do ${i.due_date})` : ''}${i.tags?.length ? ` [${i.tags.join(', ')}]` : ''}`
      ).join('\n');

  return { success: true, message: context, data: { type: 'query', context } };
}
```

**How route.ts detects query intents:**
```typescript

  // Inject into system prompt differently from mutation intents
}
```

**System prompt injection for queries (different wording than mutations):**

### Pattern 2: checklist_update Intent

The classifier learns "přidej datum K fotografovi" = update, "přidej fotografa" = create. The action-executor finds the item by ilike match (same as `completeChecklistItem`), then applies a partial update.

```typescript
// Classifier params for checklist_update:
// { title: string, updates: { due_date?: string, priority?: string, category?: string, description?: string, title?: string, tags?: string[] } }

async function updateChecklistItem(
  supabase: SupabaseClient,
  coupleId: string,
  params: { title: string; updates: Record<string, any> }
): Promise<ActionResult> {
  const { title, updates } = params;
  const { data: items } = await supabase
    .from('checklist_items')
    .select('*')
    .eq('couple_id', coupleId)
    .ilike('title', `%${title}%`);

  if (!items || items.length === 0) {
    return { success: false, message: `Nenašel jsem položku "${title}" v checklistu`, error: 'Not found' };
  }

  // Multiple match disambiguation -- return special result
  if (items.length > 1) {
    const list = items.map((it, i) => `${i + 1}) ${it.title}`).join(', ');
    return { success: false, message: `Našel jsem ${items.length} položky: ${list}. Kterou myslíš?`, error: 'Ambiguous' };
  }

  const item = items[0];
  const cleanUpdates: Record<string, any> = { ...updates };
  if (cleanUpdates.due_date) {
    cleanUpdates.due_date = parseCzechDate(cleanUpdates.due_date) ?? cleanUpdates.due_date;
  }
  // tags: replace array entirely or append? Append for INTENT-09 ("přidej tag")
  if (cleanUpdates.tags_append) {
    cleanUpdates.tags = [...(item.tags ?? []), ...cleanUpdates.tags_append];
    delete cleanUpdates.tags_append;
  }

  const { data, error } = await supabase
    .from('checklist_items')
    .update(cleanUpdates)
    .eq('id', item.id)
    .select()
    .single();

  if (error) return { success: false, message: 'Nepodařilo se aktualizovat položku', error: error.message };
  return { success: true, message: `Aktualizoval jsem "${item.title}"`, data };
}
```

### Pattern 3: budget_mark_paid with upsert fallback

```typescript
async function markBudgetPaid(
  supabase: SupabaseClient,
  coupleId: string,
  params: { name: string; amount: number }
): Promise<ActionResult> {
  const { name, amount } = params;
  const { data: items } = await supabase
    .from('budget_items')
    .select('*')
    .eq('couple_id', coupleId)
    .ilike('name', `%${name}%`);

  if (!items || items.length === 0) {
    // Create + mark paid in one step
    const { data, error } = await supabase
      .from('budget_items')
      .insert({
        couple_id: coupleId,
        name,
        category: normalizeBudgetCategory(name),
        estimated_cost: amount,
        actual_cost: amount,
        paid: true,
        source: 'ai',
      })
      .select()
      .single();
    if (error) return { success: false, message: 'Nepodařilo se vytvořit výdaj', error: error.message };
    return { success: true, message: `Vytvořil jsem a označil "${name}" jako zaplaceno (${amount.toLocaleString('cs-CZ')} Kč)`, data };
  }

  const item = items[0];
  const { data, error } = await supabase
    .from('budget_items')
    .update({ paid: true, actual_cost: amount })
    .eq('id', item.id)
    .select()
    .single();

  if (error) return { success: false, message: 'Nepodařilo se označit jako zaplaceno', error: error.message };
  return { success: true, message: `Označil jsem "${item.name}" jako zaplaceno (${amount.toLocaleString('cs-CZ')} Kč)`, data };
}
```

### Pattern 4: SearchModal (Cmd+K)

The modal needs data from all three domains simultaneously. The cleanest approach given the current architecture: make the `(dashboard)/layout.tsx` a `'use client'` provider that fetches all items once, passes them down via React context, and the SearchModal consumes that context.

However, the current layout.tsx is a server component. The simpler approach: keep layout.tsx as server component, add a new `DashboardClientWrapper` client component that receives the data as props and provides context + renders SearchModal.

**Alternative (simpler):** Pass all data directly as props to a `SearchModal` in DashboardNav. DashboardNav is already a client component. But DashboardNav doesn't have access to checklist/budget/guest data today.

**Recommended approach:** Add a `SearchModal` client component. Server-render all three data arrays in `layout.tsx`, pass them as props to a new `DashboardClientShell` client component that wraps children and renders the modal.

### Anti-Patterns to Avoid

- **Separate query API endpoint:** Query intents go through the same `/api/chat` pipeline. Don't add `/api/data-query` -- the data just needs to be in the system prompt.
- **Markdown bold in AI responses:** Must be prohibited in `buildSystemPrompt()` for query results specifically. Existing AKCE PROVEDENA block doesn't mention bold; add "Nepoužívej **bold** markup" to the query data injection section.
- **Fetch-on-every-message for query data:** Only fetch when intent IS a query intent. Not on every chat message.
- **Replacing tags instead of appending:** "přidej tag X" should append to existing tags array, not replace it. Use `tags_append` param pattern to distinguish from full tag replacement.
- **Client-side search hitting the DB:** SEARCH-01 must filter already-loaded in-memory data. No supabase calls in SearchModal.

## Common Pitfalls

### Pitfall 1: checklist_update vs checklist_add confusion in classifier
**What goes wrong:** User says "přidej datum k fotografovi" and classifier fires `checklist_add` creating a duplicate item named "fotograf" with a due_date, rather than updating the existing one.
**Why it happens:** "přidej" maps strongly to create in existing examples.
**How to avoid:** Add explicit examples that show "přidej [field] K [item]" = update. The preposition "k" (to/towards) is the linguistic signal. Also add examples with "nastav", "uprav", "změň".
**Warning signs:** Integration test where user says "přidej datum 15. 5. k fotografovi" should return `checklist_update`, not `checklist_add`.

### Pitfall 2: Multiple-match disambiguation loop
**What goes wrong:** "přidej tag urgent k fotografovi" matches 3 checklist items with "fotograf" in title. AI asks user to disambiguate. User picks "1". Now classifier sees "1" as a message and has no context.
**Why it happens:** Disambiguation requires stateful conversation context.
**How to avoid:** The existing `conversationContext` (last 3 messages) is passed to `classifyIntent`. Add a classifier rule: if previous message was a disambiguation question and current message is a number, classify as `checklist_update_confirm` with params from context. Alternatively: keep disambiguation simple -- pick the first/best match and let user correct if wrong (lower friction than a question).
**Decision needed at planning:** The CONTEXT.md says "AI se zeptá" for multi-match. This means the plan must handle the follow-up message. Simplest approach: inject previous disambiguation context into `conversationContext` so classifier sees "Haiku asked to pick 1 or 2, user said 1".

### Pitfall 3: isActionIntent() missing new intents
**What goes wrong:** New intents added to classifier but not to `isActionIntent()` or the query intent check in route.ts. Result: intents are classified but never executed.
**Why it happens:** `isActionIntent()` is a hardcoded list, easy to forget updating.
**How to avoid:** Update `isActionIntent()` and add `QUERY_INTENTS` constant in route.ts in the same task as adding the switch/case handlers.

### Pitfall 4: Query data exceeds system prompt size
**What goes wrong:** Couple has 80 checklist items + 30 budget items. Full serialization inflates system prompt by ~5KB, costs extra tokens, may hit model context limits.
**Why it happens:** "All items" injected as text.
**How to avoid:** For query intents, limit to top 15 most relevant items (pending/overdue first, then by due_date). For `status_overview`, use aggregate counts not item lists. CONTEXT.md says "max desítky položek" -- cap at 20 items per domain to be safe.

### Pitfall 5: SearchModal data staleness
**What goes wrong:** User adds a checklist item via AI chat. Switches to search modal. New item doesn't appear because search uses stale in-memory data from the server render.
**Why it happens:** Client-side search is filtering SSR-fetched data that doesn't refresh on chat mutations.
**How to avoid:** For Phase 12.1, this is an acceptable tradeoff -- document it. Full fix would require React Query or router.refresh() after every AI action. The CONTEXT.md decision says "client-side search on already-loaded data" so this known limitation is intentional.

### Pitfall 6: budget_query arithmetic
**What goes wrong:** Budget query says "zaplaceno 0 Kč" even though items have actual_cost set, because the query sums `estimated_cost` instead of `actual_cost` for paid items.
**Why it happens:** `estimated_cost` is always set; `actual_cost` is only set when paid.
**How to avoid:** In the budget query handler, compute:
- Total budget: `SUM(estimated_cost)`
- Actually spent: `SUM(actual_cost WHERE paid = true)`
- Committed (estimated, unpaid): `SUM(estimated_cost WHERE paid = false)`
- Remaining (vs total_budget from couples table): `budget_total - SUM(actual_cost WHERE paid = true)`

```typescript
function filterItems(query: string, allItems: SearchItem[]): SearchItem[] {
  if (!query.trim()) return [];
  const q = query.toLowerCase();
  return allItems.filter(item =>
    item.title.toLowerCase().includes(q) ||
    item.tags?.some(t => t.toLowerCase().includes(q)) ||
    item.subtitle?.toLowerCase().includes(q)
  );
}
```

## Open Questions

1. **Disambiguation follow-up handling**
   - What we know: CONTEXT.md says "AI se zeptá" when multiple matches found
   - What's unclear: The follow-up message ("1" or "první") needs to resolve to the original update intent -- but classifyIntent sees "1" in isolation and doesn't know what it means
   - Recommendation: During planning, decide whether to (a) pass the last disambiguation question as conversational context and add a special `disambiguation_response` intent to the classifier, or (b) simplify to "pick first match, user can correct" which avoids the state problem entirely. Option (b) is lower risk.
