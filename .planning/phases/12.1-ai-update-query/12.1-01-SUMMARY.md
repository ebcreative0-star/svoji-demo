# Phase 12.1 Plan 01: AI Update Query Summary

AI pipeline extended with update and payment intents: checklist_update (modify existing checklist items) and budget_mark_paid (mark budget items as paid or create+pay if not found), plus tag/category passthrough on all add operations.

## What Was Built

### Task 1 - Intent Classifier

- Added `checklist_update` intent: params `{ title, updates: { due_date?, priority?, category?, description?, tags_append? } }`
- Added `budget_mark_paid` intent: params `{ name, amount }`
- Extended `checklist_add` description with category guidance (vendors/venue/attire/decor/admin/ceremony mapping)
- Added `tags?: string[]` param documentation to `checklist_add` and `budget_add`
- Added 12 new disambiguation examples covering update vs create, paid vs add, tags, categories
- Added DULEZITE rules: predlozka K/k signals update, "zaplatil jsem" signals budget_mark_paid
- Updated `isActionIntent()` to include both new intents

### Task 2 - Action Executor + Route

- `updateChecklistItem`: fuzzy title search, 0-match error message, >1 disambiguation message listing all matches, 1-match update with parseCzechDate for due_date and Set-merge for tags_append
- `markBudgetPaid`: upsert pattern -- creates+marks paid if not found, updates paid+actual_cost if found while preserving estimated_cost
- Tags passthrough added to `addChecklistItem`, `addChecklistItems`, `addBudgetItem`
- Bold markup prohibition added to `buildSystemPrompt` in route.ts

## Deviations from Plan

None - plan executed exactly as written.

## Decisions Made

- `updateChecklistItem` does NOT filter by `completed` -- users should be able to update already-completed items
- `markBudgetPaid` upsert: when item not found, creates new item with category inferred via `normalizeBudgetCategory(name)` and marks paid immediately
- `tags_append` uses `Set` deduplication to avoid duplicate tags when appending
- `parseCzechDate` fallback: if parsing fails (returns null), the raw string is kept as-is -- Sonnet may have already formatted the date
- `route.ts` required only the bold markup prohibition addition; the existing pipeline already handles new intents via `isActionIntent()` + `executeAction()` switch/case
