## Summary

The AI pipeline is architecturally sound. All the machinery exists: intent classifier, action executor with all CRUD operations, Kilo gateway routing, confidence thresholds. The bugs in BUG-05 and BUG-06 are almost certainly in the intent classification prompt or the system prompt's action confirmation logic, not missing code.

BUG-07 (phantom budget item) is almost certainly caused by BudgetView.tsx silently dropping items whose category does not match any of the 12 known BUDGET_CATEGORIES values. The `groupedItems` computed value filters by `item.category === cat.value` - any item with a null, undefined, or unknown category string (e.g. AI wrote "catering" in Czech "stravovani" vs the enum "catering") will be invisible. The estimated_cost still gets summed into totalEstimated via the unfiltered `items` array, causing the phantom total.

BUG-08 (chat persistence) is subtle: the dashboard uses a Next.js App Router layout. Navigating between tabs causes the Server Component (chat/page.tsx) to re-render, which re-fetches messages from DB. But if the router caches the page, the useState in ChatInterface holds stale messages from the initial render. The fix is to either force a client-side re-fetch when the component mounts/becomes visible, or use Next.js `router.refresh()` on tab focus.

FEAT-04 (notes migration) requires a new intent type that handles multi-item, multi-category parsing in a single pass. This should be done via a dedicated Haiku call that returns structured JSON with arrays per category, not individual intent classifications per line.

**Primary recommendation:** Fix bugs first (all are 1-2 file changes each), then add batch/notes features as extensions to the existing intent pipeline. No new infrastructure needed.

## Architecture Patterns

### Current AI Pipeline (from route.ts)
```
POST /api/chat
  1. Auth + ownership check
  2. Rate limit check
  3. Load last 10 messages from DB for context
  4. classifyIntent(message, context) -> Haiku @ temp 0.1
```

### Chat Persistence Root Cause
ChatInterface.tsx uses `useState<Message[]>(initialMessages)` initialized once from props. When Next.js App Router navigates away and back:
- If page is NOT cached: Server Component re-runs, fetches fresh messages, passes as initialMessages - works correctly
- If page IS cached (router cache): Server Component does NOT re-run, old initialMessages are used, but useState already has the value - also fine
- Actual bug: messages added in current session have temp IDs (`temp-${Date.now()}`). When user navigates away and back, if server re-runs, it loads from DB where real IDs exist. The in-memory messages with temp IDs are gone. This works correctly. The reported bug may be the opposite: navigating away resets the component state entirely and the DB load (server-side, only 50 messages) may miss some edge case.

**Likely actual fix:** The chat page loads messages server-side on page load. If Next.js router cache returns a cached page without re-running the Server Component, the messages might be stale. Fix: add `export const dynamic = 'force-dynamic'` or use `cache: 'no-store'` on the Supabase query, OR add a client-side useEffect that re-fetches messages on mount.

## Common Pitfalls

### Pitfall 1: budget_add fails silently when amount is missing
**What goes wrong:** `addBudgetItem` returns `{ success: false }` when `!name || !amount`. If classifier extracts `amount: 0` or `amount: undefined`, the action fails but user sees AI say "I added it."
**Why it happens:** JS falsy check `!amount` treats `0` as missing. Also if user says "add catering" without a number, params has no amount.
**How to avoid:** In classifier prompt, add rule: budget_add REQUIRES amount. If no amount, classify as advice_request or ask for clarification. In executor, check `amount == null` not `!amount`.

### Pitfall 2: BudgetView phantom item from unknown category
**What goes wrong:** AI adds item with category not in BUDGET_CATEGORIES (e.g. "stravovani" instead of "catering"). Item exists in DB, contributes to totals, but never renders.
**How to avoid:** In action-executor.ts `addBudgetItem`, map/normalize the category to the closest BUDGET_CATEGORIES value before insert. Alternatively, add a catch-all render in BudgetView for uncategorized items.

### Pitfall 3: notes_import intent confidence collision
**What goes wrong:** A long message like "Add fotograf, catering, DJ to checklist" might trigger checklist_add with low confidence instead of batch intent.
**How to avoid:** Check message length/line count BEFORE calling classifyIntent. If message has 3+ newlines or is >300 chars, skip single-intent classification and go straight to notes_import handler.

### Pitfall 4: Chat tab-switch state loss
**What goes wrong:** User sends 3 messages, switches to budget tab, comes back - sees only the server-loaded messages (missing the last interaction if not yet saved).
**Why it happens:** Messages ARE saved to DB in route.ts. Server Component re-fetches on navigation. The issue is timing: if user navigates before the API response completes, the temp message won't be in DB yet.
**How to avoid:** This is inherent to the current architecture. The fix is to add `export const dynamic = 'force-dynamic'` to chat/page.tsx so every navigation triggers a fresh DB fetch, ensuring the latest saved messages appear.

### Pitfall 5: Summary table formatting in chat
**What goes wrong:** Markdown tables render as raw text in `whitespace-pre-wrap` div.
**Current state:** MessageBubble uses `<div className="whitespace-pre-wrap">` - no markdown rendering.
**How to avoid:** For the summary output specifically, either: (a) use plain text summary "Pridano 8 polozek: 5x checklist, 2x rozpocet, 1x host" without table syntax, or (b) add simple markdown-to-JSX for the assistant bubble (the `react-markdown` package is not installed). Simplest: use plain text with line breaks.

### Date Headers in Message List
```typescript
// In ChatInterface.tsx
import { isToday, isYesterday, isSameDay } from 'date-fns';

function MessageList({ messages }) {
  return messages.reduce((acc, msg, i) => {
    const msgDate = new Date(msg.created_at);
    const prevDate = i > 0 ? new Date(messages[i-1].created_at) : null;

    if (!prevDate || !isSameDay(msgDate, prevDate)) {
      const label = isToday(msgDate) ? 'Dnes'
        : isYesterday(msgDate) ? 'Vcera'
        : format(msgDate, 'd. MMMM', { locale: cs });
      acc.push(<DateHeader key={`header-${msg.id}`} label={label} />);
    }
    acc.push(<MessageBubble key={msg.id} message={msg} />);
    return acc;
  }, []);
}
```

## Open Questions

1. **Phantom budget item root cause: DB or UI?**
   - What we know: BudgetView groups by known categories, silently drops unknown ones. `totalEstimated` sums ALL items including ungrouped ones.
   - What's unclear: Is the 80k item actually in the DB with an unrecognized category, or is it a calculation bug (e.g., estimated_cost AND actual_cost both being summed)?
   - Recommendation: Planner should include a diagnostic step - query all budget_items for the affected couple and inspect their category values. Clean up + add catch-all render for 'other' fallback in BudgetView.

2. **notes_import: separate Haiku call or extend classifyIntent?**
   - What we know: Current classifyIntent returns single intent. Notes import needs multi-category structured output.
   - Recommendation: Separate Haiku call with notes-specific prompt. Do NOT extend classifyIntent - different output shape would break existing code.

3. **Welcome message: stored in DB or generated dynamically?**
   - Decision: dynamic generation in Server Component (fetch counts, pass as props) - no DB changes needed. This is simpler and always fresh.
